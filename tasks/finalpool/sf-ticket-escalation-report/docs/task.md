Руководителю службы поддержки нужен еженедельный аналитический отчёт об эскалациях, который выявляет заявки с риском нарушения соглашений об уровне сервиса (SLA) и отслеживает проблемы с удовлетворённостью клиентов по всей организации поддержки. Текущие правила эскалации хранятся в виде JSON-конфигурации по адресу http://localhost:30210/api/escalation_rules.json — она задаёт пороги SLA по уровням приоритета, порог удовлетворённости клиентов, ниже которого заявки помечаются, и процент предупреждения по времени ответа (response_time_warning_pct), указывающий, когда заявка приближается к пределу SLA.

Начните с чтения документа протокола эскалации (escalation_protocol.md) и файла контактов команды (team_contacts.json) в рабочем каталоге, чтобы понять текущие процедуры и список рассылки. Затем получите правила эскалации с указанного API-эндпоинта, чтобы взять конкретные пороги для анализа. Запросите данные заявок поддержки из хранилища данных ClickHouse (схема sf_data, таблица SUPPORT_CENTER__PUBLIC__TICKETS логической БД SUPPORT_CENTER) и проанализируйте все заявки относительно этих правил.

Используя процент предупреждения по времени ответа (response_time_warning_pct) из правил эскалации, определите заявки, у которых время ответа превышает эту долю от часов SLA, как заявки группы риска (at-risk). Также выявите все заявки, где удовлетворённость клиентов ниже порога satisfaction_threshold из правил. Разбейте эти результаты по уровню приоритета, по типу проблемы и по автору заявки (reporter).

Создайте Excel-файл с именем Escalation_Report.xlsx в рабочем каталоге с пятью листами. Первый лист с именем At Risk Tickets должен содержать столбцы Priority, Total_Tickets, At_Risk_Count (заявки, у которых время ответа превышает пороговую долю от часов SLA), At_Risk_Pct, Avg_Response_Hours и SLA_Hours, отсортированные по приоритету. Второй лист с именем Low Satisfaction должен содержать столбцы Priority, Issue_Type, Low_CSAT_Count (где удовлетворённость ниже порога), Total_Count, Low_CSAT_Pct и Avg_CSAT, отсортированные по Low_CSAT_Count по убыванию. Третий лист с именем Priority Summary должен содержать столбцы Priority, Total_Tickets, Low_CSAT_Tickets, Low_CSAT_Pct, Avg_Response_Hours, Avg_CSAT и SLA_Hours, отсортированные по приоритету. Четвёртый лист с именем Reporter Analysis должен содержать столбцы Reporter, Total_Tickets, Low_CSAT_Count, Low_CSAT_Pct, Avg_Response_Hours и Avg_CSAT, отсортированные по имени автора. Пятый лист с именем Summary должен содержать столбцы Metric и Value со значениями Total_Tickets, Low_Satisfaction_Tickets, Low_Satisfaction_Pct, At_Risk_Tickets, At_Risk_Pct, Avg_Response_Hours, Avg_Customer_Satisfaction, Worst_Issue_Type (тип проблемы с худшей средней удовлетворённостью), Worst_Issue_Avg_CSAT, Highest_Risk_Priority (приоритет с наибольшим процентом риска) и Highest_Risk_Pct.

Сохраняйте значения столбца Priority (Low, Medium, High, Critical) на английском языке, как в источнике данных и в JSON-правилах. Значения Reporter и Issue_Type записывайте ровно так, как они хранятся в хранилище данных.

Отправьте письмо на адрес группы эскалации (escalation_group) из файла контактов команды с темой Weekly Escalation Analysis Report. Тело письма должно кратко суммировать общее число заявок, число и процент заявок с низкой удовлетворённостью, число заявок группы риска, худший по показателям тип проблемы и пометку о том, что полный Excel-отчёт доступен для ознакомления.

Создайте страницу в Teamly с заголовком Support Escalation Analysis с содержимым, обобщающим ключевые выводы, и перечислите не менее трёх конкретных пунктов действий (action items) на основе анализа — например, сосредоточить внимание на типах проблем с самой низкой удовлетворённостью или пересмотреть уровень приоритета с самым высоким процентом риска.

Примечание: при сохранении Excel-файла через MCP-инструмент передавайте абсолютный путь к Escalation_Report.xlsx внутри рабочего каталога агента.
