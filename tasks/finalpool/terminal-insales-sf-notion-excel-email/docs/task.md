Ты аналитик по клиентскому опыту (Customer Experience) и проводишь кросс-платформенный аудит качества поддержки. У компании есть интернет-магазин (на платформе InSales) и отдельный внутренний центр поддержки (данные в ClickHouse). Руководитель направления хочет понять, где возникают системные проблемы с качеством, сопоставив данные из обеих систем.

Начни с того, что прочитай файл audit_criteria.json в своём рабочем каталоге. В нём заданы формула расчёта severity (степени серьёзности), пороги качества и контактная информация команд, которая понадобится позже.

Сначала выгрузи из магазина все заказы (orders) со статусом "refunded" или "failed". Для каждого такого заказа извлеки product IDs из line items. Затем выгрузи из магазина все отзывы на товары (product reviews) с рейтингом (rating) 2 или ниже. Это товары, которыми покупатели недовольны больше всего.

Далее запроси в системе центра поддержки все тикеты. Вычисли агрегированные метрики с группировкой по уровню приоритета (priority): общее число тикетов, среднее время отклика в часах и средний балл удовлетворённости клиентов (customer satisfaction). Также запроси в центре поддержки список агентов поддержки с их командами и уровнями навыков.

Теперь напиши Python-скрипт под названием correlate_issues.py в своём рабочем каталоге. Этот скрипт должен объединить данные магазина, чтобы выделить «проблемные товары» (problem products) — любой товар, который встречается хотя бы в одном refunded- или failed-заказе ИЛИ имеет хотя бы один отзыв с рейтингом 2 или ниже. Для каждого проблемного товара вычисли severity score по формуле из audit_criteria.json: refund_count умножить на refund_weight (30) плюс low_review_count умножить на review_weight (40). Скрипт должен выводить файл problem_products.json со списком объектов с полями product_id, product_name, category, refund_count, low_review_count и severity_score, отсортированных по severity_score по убыванию.

Напиши второй Python-скрипт под названием support_metrics.py в своём рабочем каталоге. Этот скрипт должен вычислять метрики центра поддержки по уровню приоритета: общее число тикетов, среднее время отклика в часах с округлением до двух знаков и среднюю удовлетворённость клиентов с округлением до двух знаков. Он также должен вычислять разбивку по типам проблем (issue type) с количеством тикетов и средней удовлетворённостью по каждому типу. Результат сохрани в файл support_analysis.json.

Запусти оба скрипта через терминал.

Создай книгу Excel под названием Support_Quality_Audit.xlsx в своём рабочем каталоге с четырьмя листами. Все заголовки столбцов оставь на английском, как указано ниже.

Первый лист должен называться Problem_Products со столбцами Product_ID, Product_Name, Category, Refund_Count, Low_Review_Count и Severity_Score. Включи все проблемные товары, отсортированные по Severity_Score по убыванию. Если название товара длиннее 60 символов, включай только первые 60 символов.

Второй лист должен называться Support_By_Priority со столбцами Priority, Total_Tickets, Avg_Response_Hours и Avg_Satisfaction. Включи по одной строке на каждый уровень приоритета (High, Low, Medium), отсортированные по имени приоритета в алфавитном порядке.

Третий лист должен называться Issue_Type_Breakdown со столбцами Issue_Type, Ticket_Count и Avg_Satisfaction. Включи по одной строке на каждый тип проблемы, отсортированные по Ticket_Count по убыванию.

Четвёртый лист должен называться Executive_Summary со столбцами Metric и Value. Включи следующие строки: Total Problem Products (количество всех проблемных товаров), Critical Products (количество товаров с severity score строго выше 80-го перцентиля всех значений severity score), Total Support Tickets (суммарное число тикетов по всем приоритетам), Overall Avg Satisfaction (средняя удовлетворённость по всем тикетам с округлением до двух знаков), Highest Risk Category (название категории, в которой больше всего проблемных товаров).

Оформи результаты в Teamly (наш корпоративный вики на замену Notion — база знаний). Создай пространство (space) под названием "Support Quality Tracker". Добавь в него по одной странице на каждый из топ-6 проблемных товаров (тех, у кого severity score строго выше 80-го перцентиля). На каждой странице в теле (body) укажи как минимум следующие поля: Issue — описание вида "Quality issue - " и далее краткая версия названия товара; Product — полное название товара (до 60 символов); Severity — значение "Critical"; Status — значение "Open"; Assigned_To — значение "Quality Team". Названия полей (Issue/Product/Severity/Status/Assigned_To) и их значения (Critical/Open/Quality Team) оставь на английском, как указано. Оформи страницы так, чтобы коллегам было удобно их просматривать.
